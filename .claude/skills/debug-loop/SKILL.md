# 디버깅 루프 스킬

증상에 패치를 붙이지 않는다. 근본 원인을 찾는다.

## ⚖️ Iron Law (절대 원칙)

> **조사 없이 수정하지 않는다.**
> **수정이 3번 연속 실패하면 멈추고 사용자에게 보고한다.**

수정 시도 카운터를 머릿속에 유지한다. 같은 버그에 대해 수정을 시도할 때마다
1씩 올린다. 카운터가 3에 도달하면 즉시 멈추고 아래 형식으로 보고한다:

```
🛑 3회 수정 실패 — 조사 방향 재설정 필요

시도한 수정:
1. [시도 1] — 실패 이유
2. [시도 2] — 실패 이유
3. [시도 3] — 실패 이유

현재 확인된 사실:
- ...

아직 확인하지 못한 것:
- ...

제안하는 다음 조사 방향:
- ...
```

멈추지 않고 4번째 수정을 시도하는 것은 **에러를 숨기는 행위**와 같다.

## 단계

1. **재현.** 가장 작은 확실한 재현 절차를 만든다. 재현이 안 되면 우선
   재현부터 — 재현 없는 수정은 추측이다.

2. **사실 수집.**
   - 정확한 에러 메시지(전체, 잘라내지 말 것).
   - 스택 트레이스의 첫 번째 우리 코드 프레임.
   - 입력값, 환경, 직전 변경(`git log`, `git blame`).
   - 관련 테스트의 실패 출력.

3. **가설 형성.** 가능한 원인을 3개 이내로 적는다. 각 가설에 대해
   "이 가설이 맞다면 무엇이 관찰되어야 하는가?"를 적는다.

4. **가장 저렴한 가설부터 검증.** 코드 변경 없이 검증할 수 있는 것 우선:
   로그 추가, 디버거 조사, 단위 테스트로 격리.

5. **근본 원인 확정.** 증상과 원인을 연결하는 한 줄 인과 사슬을 적는다.
   "이게 이래서 저게 저렇게 돼서 결국 X가 보인다." 사슬이 끊기면 더 파야 한다.
   사슬이 2회 이상 끊기면 2단계(사실 수집)로 돌아간다.

6. **수정.** 가장 작은 변경으로 근본 원인만 고친다. *(Iron Law 카운터 +1)*

7. **검증.** 수정 후 즉시 재현 절차를 다시 실행한다.
   - 통과 → 8단계로.
   - 실패 → Iron Law 카운터 확인. 3이면 멈추고 보고. 아니면 3단계로.

8. **회귀 테스트 추가.** 수정 전에 실패하고 수정 후에 통과해야 한다. 이게
   없으면 작업은 끝난 게 아니다.

9. **메모리 갱신.** 가정이 틀렸거나 코드베이스에 대해 비자명한 것을 배웠다면
   `mistake-recorder` 스킬로 `.claude/memory/mistakes.md` 또는 `lessons.md`에 추가한다.
   특히 Iron Law가 발동됐다면 반드시 기록 — 같은 함정에 다시 빠지지 않기 위해.

## 안티 패턴
- **조사 전에 수정부터 시도하는 것** — Iron Law 위반. 항상 2~5단계를 먼저.
- **3회 실패 후 계속 시도하는 것** — 방향이 틀렸다는 신호다. 멈추고 보고한다.
- 메시지를 보고 바로 try/except로 감싸는 것.
- 테스트가 실패하니까 어서션을 약화하는 것.
- "이상한데 다시 돌리니까 됩니다" — 그건 해결이 아니다. 플래키 원인을 진단한다.
- `print` 디버깅을 끝나고도 안 지우는 것.
- 수정 카운터를 리셋하려고 문제를 "다른 버그"로 재정의하는 것.
