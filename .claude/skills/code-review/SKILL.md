# 코드 리뷰 스킬

구조화된 리뷰를 수행한다. "괜찮아 보입니다"로 끝내지 않는다. 구체적으로,
줄 번호를 인용하고, 중요도순으로 분류한다.

## 입력
- diff(`git diff`, `git diff main...HEAD`), 파일, PR, 또는 디렉토리.
- 명확하지 않으면 어떤 범위를 볼지 묻는다.
- "보안 감사" / "security audit" 키워드가 있으면 **보안 감사 패스**를 추가로 실행한다.
- 인증·결제·사용자 데이터·파일 업로드·외부 API 연동 코드가 포함된 경우 보안 감사 패스를
  자동으로 실행하고 그 사실을 명시한다.

## 절차

1. **변경 프레이밍.** diff를 처음부터 끝까지 한 번 읽는다. *무엇을* *왜*
   바꾸는지 한 문장으로 요약한다. 못 한다면 변경이 이미 너무 산만한 것 —
   그렇게 말한다.

2. **정적 패스 (파일별).** 건드린 각 파일에 대해 다음을 점검한다:
   - 정확성: off-by-one, null/None 처리, 에러 경로, 동시성, 자원 누수(파일/소켓/락).
   - API 설계: 이름의 정확성, 파라미터 순서, 기본값, 호환성을 깨는 변경.
   - 보안: 인젝션, 역직렬화, 경로 트래버설, 비밀 노출, 미검증 입력 — `.claude/rules/20-security.md` 참고.
     보안 민감 코드가 감지되면 아래 **보안 감사 패스** 실행.
   - 성능: 명백한 N+1, 우발적 2차 복잡도, 핫패스 할당.
   - 테스트 가능성: 새 코드가 테스트에서 도달 가능한가?

3. **테스트 패스.** 새 테스트가 있는가? 변경을 커버하는가? 기존 테스트는
   여전히 통과하는가? 아직 안 돌렸다면 지금 돌린다.

4. **스타일 패스.** 주변 코드와 `.claude/rules/10-coding-standards.md`와
   비교한다. 스타일 nit은 별도 섹션에 둔다 — 진짜 발견과 섞지 않는다.

5. **메모리 패스.** `.claude/memory/mistakes.md`와 `lessons.md`를 훑는다.
   diff가 알려진 실수를 반복한다면 **HIGH**로 표시하고 항목을 링크한다.

## 🔐 보안 감사 패스 (심층)

> 이 패스는 명시적 요청 또는 보안 민감 코드 감지 시 자동 실행된다.
> **신뢰도 기준: 8/10 이상 확신이 없으면 발견으로 보고하지 않는다.**
> 모든 발견에는 반드시 **구체적인 공격 시나리오**를 붙인다. "가능성 있음"은 발견이 아니다.

### OWASP Top 10 체크리스트

각 항목은 변경된 코드에 해당되는 것만 점검한다. 해당 없는 항목은 건너뛴다.

| # | 카테고리 | 점검 포인트 |
|---|---|---|
| A01 | **접근 제어 실패** | 인가 검사 누락, IDOR(직접 객체 참조), 수평/수직 권한 상승, 관리자 기능 노출 |
| A02 | **암호화 실패** | 평문 민감 데이터, 약한 알고리즘(MD5/SHA1), 하드코딩된 키/비밀번호, TLS 미적용 |
| A03 | **인젝션** | SQL·NoSQL·명령어·LDAP·XPath 인젝션, 파라미터화 쿼리 미사용, 셸 문자열 조립 |
| A04 | **안전하지 않은 설계** | 비즈니스 로직 결함, rate limiting 부재, 취약한 비밀번호 정책, 2FA 우회 가능 |
| A05 | **보안 설정 오류** | 디버그 모드 활성, 기본 자격증명, 불필요한 기능 활성, CORS 과잉 허용, 스택 트레이스 노출 |
| A06 | **취약한 컴포넌트** | 알려진 취약점 있는 버전 고정, 라이선스 문제, 유지보수 중단 패키지 |
| A07 | **인증 및 세션 관리 실패** | 세션 고정, 약한 토큰 엔트로피, 만료 없는 토큰, 로그아웃 시 세션 미삭제 |
| A08 | **소프트웨어·데이터 무결성 실패** | 서명 미검증 업데이트, 신뢰할 수 없는 소스 역직렬화, CI/CD 파이프라인 인젝션 |
| A09 | **보안 로깅·모니터링 실패** | 인증 실패 미기록, 로그에 PII/비밀 포함, 감사 이벤트 누락 |
| A10 | **SSRF** | 사용자 제공 URL 패치, 내부 메타데이터 엔드포인트 접근, URL 화이트리스트 미적용 |

### 발견 형식 (보안)

```
🔐 [A0N — 카테고리명] [파일:줄]
- 문제: <무엇이 잘못됐나>
- 공격 시나리오: <구체적으로 어떻게 악용되는가 — 실제 페이로드 또는 단계>
- 영향: <데이터 유출 / 권한 상승 / 서비스 중단 / ...>
- 수정: <최소 변경으로 해결하는 방법>
- 신뢰도: N/10
```

신뢰도 7 이하는 "참고" 섹션에 별도 분리하고 발견 사항 카운트에서 제외한다.

### 자동 제외 (False Positive 방지)

아래는 코드에 있어도 발견으로 보고하지 않는다:

- 테스트 파일의 하드코딩된 더미 자격증명 (`test_password`, `fake_api_key` 등)
- 환경 변수로 주입되는 설정값 (`os.getenv(...)`, `process.env.XXX`)
- 이미 상위 레이어에서 처리된 인증 (미들웨어, 데코레이터로 보호된 라우트)
- `localhost` / `127.0.0.1` 전용 엔드포인트
- 문서·주석 안의 예시 코드
- 이미 `20-security.md`에 인지된 설계 결정으로 기록된 패턴

## 출력 형식

```
## 요약
<한 단락>

## 발견 사항

### 🔴 차단 (Blocking)
- [파일:줄] <문제> — <왜 중요한가> — <제안된 수정>

### 🟡 고치는 게 좋음
- ...

### 🟢 사소한 것 (nits)
- ...

## 🔐 보안 감사 결과  ← 보안 감사 패스 실행 시에만 포함
### 발견된 취약점 (신뢰도 8/10+)
- <발견 형식 참고>

### 참고 (신뢰도 7/10 이하)
- <낮은 신뢰도 항목>

### 점검 범위
- 실행된 OWASP 항목: A01, A03, A07 (해당 코드 기준)
- 건너뛴 항목: A06 (의존성 변경 없음), ...

## 테스트
- 실행: <명령>
- 결과: <성공/실패/미실행>
- 변경 커버리지: <좋음 / 부분 / 없음>

## 결론
<승인 / 변경 요청 / 논의 필요>
```

## 규칙
- 끝까지 읽지 않은 변경을 승인하지 않는다.
- 줄 번호를 지어내지 않는다. 인용했다면 읽은 것이다.
- 발견 사항이 0이라면 한 번 더 본다. 코드보다 리뷰가 더 자주 틀린다.
- 보안 발견은 신뢰도 8/10 미만이면 보고하지 않는다. 노이즈는 진짜 위협을 묻는다.
- 공격 시나리오 없는 보안 발견은 발견이 아니다.
