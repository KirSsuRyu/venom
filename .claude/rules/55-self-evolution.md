# 55 — 자기 진화 프로토콜

> Venom은 1회성 도구가 아니다. **살아있는 심비오트**다.
> 프로젝트에서 일하면서 끊임없이 배우고, 스스로의 규칙·스킬·훅을 진화시킨다.
> `/venom`이 최초 흡수라면, 이 프로토콜은 **매일의 호흡**이다.

## 진화의 4원칙

1. **실수에서 규칙이 태어난다** — 같은 실수가 2번 기록되면 그것을 막는
   규칙 또는 hook이 아직 없다는 뜻이다. 만들어야 한다.
2. **반복에서 스킬이 태어난다** — 같은 작업 패턴을 3번 이상 수행하면
   그것은 스킬로 응결되어야 한다.
3. **발견에서 메모리가 자란다** — 코드베이스의 비자명한 사실을 알게 되면
   즉시 기록한다. 다음 세션의 나는 오늘의 나보다 더 많이 알아야 한다.
4. **승급하면 소멸한다** — mistakes.md/lessons.md 항목이 규칙·스킬·hook으로
   승급되면 decisions.md에 흡수된 뒤 원본은 삭제된다.
   메모리는 "아직 진화 안 된 것"만 담는다. 진화된 것은 규칙이 기억한다.

---

## 진화 트리거: 언제 진화하는가

### 자동 트리거 (이 조건이 감지되면 즉시 진화 행동)

| 조건 | 진화 행동 |
|---|---|
| `mistakes.md`에 동일 태그 실수 2회 이상 | 해당 영역의 규칙 강화 또는 hook 신규 생성 |
| 같은 패턴의 코드를 3회 이상 작성 | 해당 패턴을 스킬로 추출 |
| 사용자가 "이거 항상 이렇게 해" 류의 교정 | 해당 관례를 규칙에 추가 |
| hook이 차단했는데 이유가 불분명 | hook 메시지 개선 또는 규칙에 설명 추가 |
| 새 외부 서비스/도구/라이브러리 도입 | `lessons.md`에 사용법 기록, 필요시 규칙/스킬 추가 |
| 아키텍처 경계가 변경됨 | `61-architecture.md` 갱신, `decisions.md`에 ADR |
| 테스트/빌드 명령이 변경됨 | `60-project.md` 및 `project-build` 스킬 갱신 |

### 수동 트리거 (사용자가 요청)

- **"이거 규칙으로 만들어"** → 해당 관례를 적절한 규칙 파일에 추가
- **"이 작업 스킬로 만들어"** → 반복 가능한 절차를 스킬로 추출
- **"이거 차단해"** → hook에 패턴 추가 또는 신규 hook 생성
- **"기억해"** → `lessons.md` 또는 `decisions.md`에 기록

## 진화 절차

### 1. 규칙 진화 (rules/)

**파일 배치 규칙:**
- **기존 기반 규칙** (`00-` ~ `55-` 번호 prefix): 기존 flat 파일 그대로 보강
- **진화로 새로 생성되는 규칙**: `evolved/` 폴더에 배치
  ```
  .claude/rules/evolved/<이름>.md
  ```

**추가할 때:**
```
1. 진화 이유를 한 줄로 적는다 (어떤 실수/관례에서 비롯되었나)
2. 기존 규칙 파일에 추가할 수 있으면 추가한다 (새 파일보다 기존 파일 보강 우선)
3. 새 규칙 파일이 필요하면:
   - .claude/rules/evolved/<이름>.md 로 생성
   - CLAUDE.md 부록 테이블에 등재
4. decisions.md에 ADR로 남긴다
```

**수정할 때:**
```
1. 기존 규칙이 현실과 맞지 않는다면 (프로젝트가 변했거나 규칙이 틀렸거나)
2. 원본을 읽고, 무엇이 틀렸는지 파악한다
3. 수정하고, decisions.md에 왜 바꿨는지 ADR로 남긴다
```

**삭제할 때:**
```
1. 규칙을 삭제하기 전에 사용자에게 묻는다
2. 삭제 대신 "폐기됨" 표시를 우선 고려한다
3. decisions.md에 폐기 이유를 ADR로 남긴다
```

### 2. 스킬 진화 (skills/)

**파일 배치 규칙:**
- **기존 기반 스킬**: 기존 `skills/<name>/` 그대로 유지
- **진화로 새로 생성되는 스킬**: `evolved/` 폴더에 배치
  ```
  .claude/skills/evolved/<name>/SKILL.md
  ```

**새 스킬 추출 기준:**
- 3회 이상 반복된 작업 패턴
- 5단계 이상의 절차가 있는 작업
- 실수 가능성이 높은 작업 (순서가 중요하거나, 누락하기 쉬운 단계가 있거나)

**스킬 형식:**
```markdown
---
name: <스킬-이름>
description: <언제 트리거하는가, 한 줄>
---
# <스킬 제목>
<왜 이 스킬이 필요한가, 한 줄>
## 절차
1. ...
## 안티 패턴
- ...
```

**기존 스킬 진화:**
- 실제 사용 중 빠진 단계를 발견하면 즉시 추가
- 불필요한 단계를 발견하면 정리
- 프로젝트 특화 컨텍스트가 생기면 보강

### 3. 서브에이전트 진화 (agents/)

**파일 배치 규칙:**
- **기존 기반 에이전트** (`code-reviewer`, `debug-detective`, `test-writer`, `security-auditor`): 기존 flat 파일 유지
- **진화로 새로 생성되는 에이전트**: `evolved/` 폴더에 배치
  ```
  .claude/agents/evolved/<name>.md
  ```

**새 에이전트 추출 기준:**
- 특정 도메인 작업이 **메인 컨텍스트를 5% 이상 잠식**하는 것을 3회 이상 관찰
  (예: 반복되는 DB 마이그레이션 검증, API 계약 리뷰, 로컬라이제이션 검토)
- 작업이 독립적이어서 메인 대화 없이도 완결 가능한가?
- 격리 실행 시 토큰 절약이 분명한가? (격리 에이전트는 자기 컨텍스트 소모하므로 극소 작업엔 과함)

**에이전트 형식:**
```markdown
---
name: <kebab-case-name>
description: <언제 라우팅되나, 2~3문장. 트리거 단서와 반환 형식 명시>
tools: Read, Grep, Glob, Bash  # 최소 권한
---
# <에이전트 제목>
<역할 한 줄>

## 절차
...

## 규칙
- 파일을 쓰는가/읽기만 하는가 명시
- 격리 컨텍스트임을 이용해 호출자 상태를 추측하지 않는다
- 최종 보고 형식 고정
```

**에이전트 vs 스킬 판정:**
- 스킬: 메인 컨텍스트에서 절차를 안내 — 대화 맥락이 필요할 때.
- 서브에이전트: 격리된 컨텍스트에서 작업을 위임 — 메인 토큰 절약이 필요할 때.
- 규칙 하나만 추가하면 되는 것은 에이전트로 만들지 않는다.

### 4. Hook 진화 (hooks/)

**새 hook 생성 기준:**
- 규칙만으로는 방어가 안 되는 위험 행동 (규칙은 조언, hook은 강제)
- 같은 실수가 규칙 추가 후에도 반복될 때
- 자동화할 수 있는 검증 단계

**hook 파일 배치 규칙:**
- **기존 기반 훅** (`session-start.sh` 등): 기존 flat 파일 유지
- **진화로 새로 생성되는 훅**: `evolved/` 폴더에 배치
  ```
  .claude/hooks/evolved/<이름>.sh
  ```

**hook 생성 절차:**
```
1. .claude/hooks/evolved/<이름>.sh 작성
   - #!/usr/bin/env bash
   - set -euo pipefail
2. bash -n으로 문법 검증
3. chmod +x 부여
4. settings.json의 적절한 이벤트에 등록 (서브폴더 경로 그대로 사용)
5. JSON 유효성 검증:
   python3 -c "import json; json.load(open('.claude/settings.json'))"
6. 대표 입력으로 스모크 테스트
7. decisions.md에 ADR로 남김
```

### 5. 메모리 진화 (memory/)

**매 세션 끝에 자문한다:**
- 이 세션에서 코드베이스에 대해 새로 알게 된 게 있는가? → `lessons.md`
- 실수를 했는가? → `mistakes.md` (hook이 자동 기록, 교훈은 내가 채움)
- 아키텍처 결정을 했는가? → `decisions.md`
- 기존 메모리 항목이 틀렸거나 오래됐는가? → 갱신 또는 폐기 표시

**진화 완료 후 메모리 정리:**

진화(규칙/스킬/hook 생성)가 완료되면 반드시:
```
1. decisions.md에 ADR 추가 — 기원 필드에 삭제할 항목 명시
2. 해당 mistakes.md 또는 lessons.md 항목 삭제
```

이 정리 단계를 빠뜨리면 `mistakes.md`에 "이미 막힌 실수"가 계속 쌓여
세션 로드 시 불필요한 컨텍스트가 낭비된다.

## 진화의 위생 규칙

1. **작게 진화한다.** 한 번에 규칙 하나, 스킬 하나, hook 하나.
   한꺼번에 여러 개를 만들면 어떤 변경이 문제인지 추적이 안 된다.
2. **이유를 남긴다.** 모든 진화에는 "왜"가 있어야 한다.
   이유 없는 규칙은 다음 세션에서 의문의 쓰레기가 된다.
3. **검증한다.** 새 hook은 반드시 문법 검증 + 스모크 테스트.
   새 규칙은 실제 상황에 적용해보고 맞는지 확인.
4. **커밋은 사용자가 한다.** 진화 결과를 만들되 커밋은 사용자에게 맡긴다.
5. **비밀을 기록하지 않는다.** 진화 과정에서 알게 된 비밀 값은
   변수명과 구조만 기록하고 값은 절대 기록하지 않는다.
6. **과잉 진화를 경계한다.** 모든 관찰이 규칙이 될 필요는 없다.
   2번 이상 반복되지 않는 것은 메모리에만 남기고 규칙화하지 않는다.
7. **진화 후 반드시 정리한다.** 규칙/스킬/hook 생성이 끝나면
   원본 mistakes.md/lessons.md 항목을 삭제한다. 정리 없는 진화는 반쪽짜리다.

---

## 진화 보고

규칙·스킬·hook을 진화시켰으면 사용자에게 간단히 보고한다:

```
📝 진화: <무엇을> <어떻게> 했습니다.
- 이유: <왜>
- 파일: <경로>
- 검증: <어떻게 확인했는지>
```

이렇게 하면 사용자가 진화를 인지하고, 필요하면 되돌릴 수 있다.
